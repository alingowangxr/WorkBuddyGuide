# 第 26 章 崗位路線圖：不同崗位如何把 WorkBuddy 用深

崗位應用不應從“這個崗位有哪些提示詞”開始，而應從三個問題開始：哪些工作重複發生，哪些產物有穩定標準，哪些決定必須由人承擔責任。

## 崗位應用的四級成熟度

| 等級 | 工作方式 | 典型形態 | 進入下一級的條件 |
|-|-|-|-|
| L1 單次輔助 | 人給完整材料，AI 生成草稿 | 紀要、摘要、表格、PPT | 輸出可驗收，連續成功 3 次 |
| L2 固定模板 | 使用任務卡和標準輸出 | 週報、覆盤、客戶簡報 | 輸入欄位和驗收規則穩定 |
| L3 工作流 | Skill、資料庫、聯結器協作 | 內容流、知識流、報表流 | 有負責人、日誌和異常處理 |
| L4 團隊能力 | 專案、專家團和自動化執行 | 部門級數字員工 | 許可權、成本和資產治理到位 |

每個崗位都應從 L1 的低風險任務開始。不是所有任務都需要進入 L4；低頻、差異大或責任重的工作，保持“AI 輔助 + 人工決策”更合理。

## 產品經理

| 場景 | 輸入 | 交付物 |
|-|-|-|
| 使用者訪談整理 | 錄音轉寫、訪談提綱 | 痛點、原話、待驗證問題 |
| 反饋聚類與版本覆盤 | 工單、評論、埋點摘要 | 問題分類、影響範圍、證據表 |
| 競品證據庫 | 官網、定價、更新日誌、評論 | 可追溯對比表和定期簡報 |
| 產品發現工作流 | 反饋、資料、競品、業務目標 | 機會地圖和決策材料 |

## 銷售與售前

| 場景 | 交付物 | 風險點 |
|-|-|-|
| 會後紀要 | 需求、異議、待辦、負責人 | 把推測寫成客戶承諾 |
| 客戶研究 | 畫像、公開動態、機會假設 | 使用過時或同名公司資料 |
| 方案與演示 | 方案大綱、演示指令碼、FAQ | 虛構能力、價格和案例 |
| 跟進工作流 | CRM 草稿、提醒、材料更新 | 未確認寫入和外發 |

## HR 與行政

| 場景 | 推薦做法 | 必須控制 |
|-|-|-|
| 制度摘要、材料檢查 | 基於批准檔案生成 FAQ | 版本日期和適用範圍 |
| 入職與培訓 | 生成學習路徑、課件和清單 | 員工資料最小化 |
| HR 知識助手 | 連線制度庫回答常見問題 | 許可權、引用與升級人工 |
| 人事流程輔助 | 提醒、材料完整性、報表 | 不自動做僱傭決策 |

## 財務與法務

| 財務場景 | 法務場景 | 人工關口 |
|-|-|-|
| 發票識別、賬單整理 | 條款提取、版本對照 | 金額與原文複核 |
| 預算差異、異常清單 | 標準條款偏離清單 | 口徑和風險判斷 |
| 月結材料流水線 | 合同知識庫、審閱工作臺 | 專業人員簽字 |
| 受控系統回寫 | 合同流程提醒 | 付款、申報、簽署確認 |

## 研發與 IT

| 場景 | 交付物 | 檢查點 |
|-|-|-|
| 程式碼理解、技術文件 | 架構圖、README、模組清單 | 與實際程式碼一致 |
| 測試與缺陷分析 | 測試矩陣、復現步驟、修復建議 | 測試通過、程式碼評審 |
| 開發工作流 | 分支改動、CI 診斷、釋出說明 | 合併和部署確認 |
| IT 自動化 | 巡檢、告警摘要、知識庫 | 生產許可權和事故響應 |

## 部門推廣模板

每個崗位先選一個場景，填寫下面這張卡：

| 專案 | 填寫內容 |
|-|-|
| 原流程 | 誰在什麼時間，用哪些系統，花多久 |
| 輸入 | 檔案、欄位、許可權、質量問題 |
| 輸出 | 檔案、系統記錄、通知物件 |
| 驗收 | 準確性、完整性、時效和風險 |
| 人工關口 | 誰確認，什麼情況下必須暫停 |
| 指標 | 基線、目標值、觀察週期 |
| 負責人 | 業務、技術、資產維護人 |

崗位落地的終點不是“每個人都會問 AI”，而是形成一批能被新人複用、由負責人維護、出了問題能追溯的標準工作流。
