# 參與共創

WorkBuddy 的能力、介面和使用方式會持續變化。這本藍皮書希望由真實使用者共同維護：不僅糾正事實，也不斷補充可復現的任務、失敗經驗和團隊方法。

## 可以貢獻什麼

- 修正錯別字、失效連結、過時介面和不準確表述。
- 補充一個真實、可復現的 WorkBuddy 案例。
- 為現有案例增加輸入、輸出、驗收標準和失敗處理。
- 補充崗位或行業工作流。
- 改進截圖、圖表、導航和無障礙體驗。

## 提交社群 Case

如果你已經用 WorkBuddy 跑通了一項真實任務，推薦優先提交到[社群案例集](/cases/)。每個 Case 需要說明使用場景、使用的 Skills、在 WorkBuddy 中的操作、最終效果和驗收標準。

請先閱讀[Case 投稿指南](/community/case-contributing)。PR 合併後，案例會自動顯示在網站的案例集合頁；經過進一步復現和編輯的經典任務，未來可能進入藍皮書正文。

## 最輕量的方式：提交 Issue

如果暫時不想修改檔案，可以在 [GitHub Issues](https://github.com/AlephAITech/WorkBuddyGuide/issues) 中說明：

1. 哪一頁需要調整。
2. 現在的問題是什麼。
3. 建議怎樣修改。
4. 如果涉及產品事實，請附上來源或截圖。

## 直接修改內容

1. Fork 或克隆倉庫。
2. 找到 `docs/bluebook/` 下對應章節的 `index.md`。
3. 修改內容或在同目錄的 `assets/` 中補充素材。
4. 本地執行 `npm install` 和 `npm run docs:build`。
5. 提交 Pull Request，並說明改動、原因和驗證方式。

完整規範見倉庫根目錄的 [CONTRIBUTING.md](https://github.com/AlephAITech/WorkBuddyGuide/blob/main/CONTRIBUTING.md)。

## 內容原則

- **真實任務優先**：說明輸入、步驟、交付物和驗收方式。
- **事實與經驗分開**：產品事實儘量附來源；個人經驗說明適用條件。
- **對新手友好**：不要假設讀者知道工具、許可權和術語。
- **安全邊界清晰**：涉及檔案、賬號、金鑰、釋出和外部操作時說明風險。
- **尊重隱私與版權**：不要提交金鑰、個人資料或無權公開的資料。
