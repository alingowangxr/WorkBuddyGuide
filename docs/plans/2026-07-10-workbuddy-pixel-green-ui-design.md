# WorkBuddy Guide 官網 UI 重設計方案

> 目標：把當前偏 VitePress 預設風格的文件站，升級為具有明確品牌識別度的「高階編輯型 AI 知識站」。視覺參考使用者提供的熒光黃綠色畫素海報，但不把整站做成復古遊戲介面。

## 1. 結論先行

推薦方向：**Editorial Pixel / 編輯型畫素科技感**。

- 用米白、白色和墨黑承擔長文閱讀與專業感。
- 用熒光黃綠色建立品牌記憶，只出現在 Hero、主要按鈕、當前狀態和少量強調區。
- 畫素元素控制在整頁視覺的 10%–15%，主要用於小圖示、章節編號、裝飾卡片和微動效。
- 中文標題和正文堅持現代無襯線字型；畫素字型只用於英文標籤、數字、狀態碼，不能用於長段中文。
- 保留現有 VitePress 的搜尋、側邊欄、目錄、深色模式、Markdown 內容和全部路由，不重做技術架構。

最終應該讓人感到：**這是一本來自 AI Agent 時代的開源工作手冊，而不是一個套了綠色主題的預設文件站。**

---

## 2. 對當前站點的判斷

### 已有優勢

- 首頁的資訊順序清楚：品牌主張 → 三個價值點 → 四段閱讀路線。
- 文件站核心能力完整：本地搜尋、側邊欄、本頁目錄、上一篇/下一篇、深色模式。
- 移動端基礎結構可用，現有內容和路由不需要重寫。
- VitePress 預設主題適合繼續擴充套件，沒有必要換框架。

### 主要問題

- 紫色漸變、圓角卡片、點陣背景都很常見，缺少 WorkBuddy 自己的品牌特徵。
- 首頁 Hero 左側資訊堆疊，右側大面積留白，沒有形成一個能被截圖傳播的主視覺。
- 三張功能卡和四張閱讀卡視覺權重接近，頁面缺少強弱節奏。
- 章節頁幾乎是預設文件站樣式，側邊欄、正文、目錄之間缺少精緻的層級關係。
- 圓角、漸變和陰影偏柔，和參考圖的清脆畫素感不一致。
- 當前畫素感只存在於點陣背景，並沒有變成統一的品牌語言。

## 3. 三個可選方向

| 方向 | 視覺特徵 | 優點 | 風險 | 建議 |
| --- | --- | --- | --- | --- |
| A. 全幅畫素海報 | 大面積熒光綠、超大畫素字、強烈黑白對比 | 傳播性最強，第一眼最像參考圖 | 長文閱讀容易疲勞，也容易像活動頁或遊戲站 | 不推薦整站採用 |
| B. 編輯型畫素科技感 | 米白閱讀底色、熒光綠品牌區、現代排版、畫素細節 | 兼顧高階感、識別度和閱讀體驗 | 對留白、字型層級和素材質量要求高 | **推薦** |
| C. 暗色命令中心 | 石墨黑底、熒光綠狀態、終端/控制台感 | 科技感強，適合開發者 | 不符合藍皮書的長期閱讀場景，中文密度高時壓迫 | 只作為深色模式參考 |

執行時直接採用 B，不需要再混合三套風格。

## 4. 品牌關鍵詞與設計原則

### 品牌關鍵詞

**開源、可靠、行動、系統、協作、未來感、可復現。**

### 五條設計原則

1. **先像一本好讀的書，再像一個有個性的科技品牌。**
2. **畫素只做記憶點，不做閱讀負擔。**
3. **綠色負責聚焦，黑白負責秩序。**
4. **減少柔和漸變和膠囊按鈕，增加清晰邊界和硬朗結構。**
5. **所有裝飾都要服務於章節、路徑或狀態，不增加無意義噪聲。**

## 5. 視覺系統

### 5.1 色彩 Token

#### 淺色模式

| 用途 | 色值 | 使用規則 |
| --- | --- | --- |
| Brand Acid | `#D8F238` | Hero、主按鈕、活動狀態、大面積品牌塊 |
| Brand Acid Strong | `#BCE52E` | Hover、圖表強調、深色模式中的次級品牌綠 |
| Ink | `#12140F` | 標題、主要按鈕、深色區域背景 |
| Paper | `#F5F7F0` | 頁面主背景，輕微偏暖，不用冷白灰 |
| Surface | `#FFFFFF` | 卡片、正文、搜尋框 |
| Soft Green | `#EEF6D1` | 提示框、引用、淺強調區 |
| Text Secondary | `#62675E` | 正文次級資訊、描述 |
| Line | `#D9DDD2` | 分隔線、卡片邊框、表格線 |
| Accessible Green | `#355E18` | 白底上的綠色文字和連結狀態 |

#### 深色模式

| 用途 | 色值 |
| --- | --- |
| Background | `#0F110D` |
| Surface | `#181B15` |
| Surface Raised | `#20241C` |
| Text Primary | `#F3F5ED` |
| Text Secondary | `#AEB4A7` |
| Line | `#33392E` |
| Brand Acid | `#D8F238` |

#### 禁止事項

- 不用熒光綠作為白底小字號正文，因為對比度不足。
- 不再使用紫藍漸變作為品牌色。
- 不用大面積綠色漸變；品牌綠儘量保持純色和平面感。
- 同一屏最多一個大面積綠色區域。

### 5.2 字型與排版

- 中文標題：`PingFang SC / Microsoft YaHei / system-ui`，字重 700–800。
- 中文正文：系統無襯線，字重 400–500，章節正文行高 `1.8–1.9`。
- 英文與數字：優先 `Inter Tight / Inter / system-ui`。
- 畫素字型：只用於 `PART 01`、`OPEN SOURCE`、`27 CHAPTERS`、編號和裝飾詞。可選擇一款許可證清晰、支援英文和數字的畫素字型，並本地託管。
- 不讓畫素字型承擔中文 H1、導航或正文。

建議字號：

| 場景 | 桌面端 | 移動端 |
| --- | --- | --- |
| 首頁 H1 | `72–88px / 0.98` | `42–50px / 1.05` |
| 首頁副標題 | `20px / 1.7` | `17px / 1.7` |
| 區塊標題 | `40–48px` | `30–36px` |
| 章節 H1 | `44–52px` | `34–40px` |
| 正文 | `17–18px` | `16–17px` |
| 畫素標籤 | `12–14px` | `11–12px` |

### 5.3 圓角、邊框和陰影

- Hero 大容器：`8px`，不要誇張大圓角。
- 普通卡片：`6–8px`。
- 按鈕：`4–6px`，取消膠囊形態。
- 邊框：`1px` 或 `1.5px`，使用 Ink/Line 色。
- 重點卡片 Hover：使用 `4px 4px 0` 的剋制硬陰影，避免柔軟大模糊陰影。
- 正文圖片可以保留 `8px` 圓角，但不要每一層容器都圓角。

### 5.4 畫素資產規範

需要一套真實素材，不要用 emoji、文本符號、臨時 CSS 圖形或隨手畫的線性圖示代替。

建議準備 6 個單色 1-bit 畫素圖示：

1. 工作任務 / Cursor
2. AI Agent / Buddy
3. Skill / 工具箱
4. Workflow / 節點連線
5. Team / 多角色協作
6. Book / 開源藍皮書

資產要求：

- 黑色主體、透明背景，基於 `32×32` 或 `48×48` 網格繪製。
- 輸出 4 倍 PNG 或許可證清晰的 SVG；PNG 展示時啟用畫素化渲染，避免模糊。
- Hero 中將圖示放進白色方卡，允許 `±3deg` 的輕微旋轉，模擬參考圖的貼紙感。
- 額外準備一張白色小方塊點綴圖案，用作 Hero 邊緣裝飾；不要全站重複鋪滿。
- 圖示必須語義清楚，避免骷髏、手柄、宇宙飛船等與產品無關的遊戲化符號。

## 6. 首頁結構方案

### 6.1 頂部導航

- 高度約 `64–68px`，白色/米白底，底部 1px 分隔線。
- 左側為 WorkBuddy Guide 字標 + 小型畫素書本標誌。
- 中部或右側保留：`首頁 / 開始閱讀 / 閱讀指南 / 參與共創`。
- 搜尋入口從左側移到右側工具區，和深色模式、GitHub 同組。
- 當前導航項不再用紫色文字，改成 Ink 字 + 下方 3px 熒光綠短線或綠色畫素方塊。
- 移動端只保留字標、搜尋、選單；點選區域至少 `44×44px`。

### 6.2 Hero：整站最重要的品牌畫面

Hero 使用一個居中的熒光綠大色塊，桌面端最大寬度約 `1280px`、高度約 `540–600px`，內容分成 7:5 兩列。

左側內容：

```text
OPEN-SOURCE · 27 CHAPTERS · 2026

從第一項任務，
到一支 AI 團隊

一套以真實工作為主線的 WorkBuddy 實踐路徑。
先用起來，再把一次成功沉澱為可複用的工作系統。

[ 開始閱讀 → ]  [ 檢視閱讀路線 ]
```

右側內容：

- 3–4 張白色畫素圖示卡片，以錯位、輕旋轉方式懸浮在綠色區域內。
- 中間可以出現一個大的 `WB_` 或 `WORK / SYSTEM` 畫素標籤，但不要覆蓋中文主標題。
- 右下角放 `27 CHAPTERS / 4 PARTS / ∞ WORKFLOWS` 三個數字資訊，作為可信度和內容規模說明。

Hero 視覺要求：

- 中文主標題使用現代粗體，不用畫素字型。
- 主按鈕用 Ink 黑底 + 白字；次按鈕用透明底 + Ink 邊框。
- 不新增玻璃擬態、彩色光暈、3D 球體、人物 AI 插畫。
- 首屏必須在 1440×900 下完整看到標題、CTA 和至少兩張畫素卡片。

### 6.3 價值條

緊接 Hero 放一條黑底靜態資訊帶，不做無限滾動跑馬燈：

`真實任務 REAL TASKS  ·  可復現 REPRODUCIBLE  ·  社群共創 OPEN SOURCE  ·  系統沉澱 WORK SYSTEM`

作用是把視覺從綠色 Hero 過渡到米白內容區，並快速傳達專案差異。

### 6.4 四段閱讀路線

- 保留現有四段內容，但改成更像編輯排版的 2×2 大卡片。
- 每張卡左上是畫素標籤 `PART 01`，右上是章節範圍，如 `CH. 01–10`。
- 標題 28–32px，描述最多兩行。
- 卡片底部增加一條視覺化路徑線，Hover 時由灰色變成熒光綠。
- PART 01 作為推薦入口，可用淺綠色底；其他卡片保持白底。
- 四張卡不要都用相同漸變。

### 6.5 按任務進入

增加一個實用型發現區，用已有章節生成 6 個入口，不增加新頁面：

- 辦公文件
- 檔案與遠端控制
- 資訊與知識管理
- 投資與專業分析
- 影片與內容增長
- Skill、多 Agent 與自動化

每個入口由畫素圖示、標題、2–3 個關鍵片語成。桌面端 3×2，移動端橫向可滾動或單列。這個區塊的目的不是裝飾，而是幫助“帶著具體任務來的讀者”更快進入對應章節。

### 6.6 方法路徑

用一條簡潔的四步路徑說明藍皮書的方法論：

`TASK → CASE → WORKFLOW → AI TEAM`

中文解釋：

1. 完成一個真實任務
2. 覆盤並形成可復現案例
3. 沉澱為 Skill 和自動化工作流
4. 組合成可協作的 AI 團隊

使用真實畫素節點/箭頭資產或許可證清晰的圖示，不用臨時 CSS 繪製複雜插畫。

### 6.7 共創 CTA 與頁尾

- 最後一個 CTA 使用 Ink 黑底，左側大標題 `這本藍皮書，也可以由你繼續寫下去。`
- 右側兩個按鈕：`參與共創`、`前往 GitHub`。
- 放一個熒光綠的畫素書本圖示作為視覺錨點。
- 頁尾保持簡潔，包含專案宣告、GitHub、閱讀指南和版權。

## 7. 文件閱讀頁方案

文件頁不需要像首頁一樣高飽和，目標是“安靜、清晰、具有品牌細節”。

### 7.1 整體佈局

- 桌面端繼續使用左側欄 + 正文 + 右側目錄三欄結構。
- 左側欄約 `272–288px`，背景使用 Paper，右邊 1px 分隔線。
- 正文最佳閱讀寬度控制在 `720–780px`。
- 右側目錄約 `200px`，使用一條細豎線連線層級。

### 7.2 側邊欄

- 篇章標題使用小號畫素英文標籤 + 中文標題，例如 `PART 01 / 使用手冊`。
- 當前章節使用 Ink 黑底、白字，並在左側加 6px 熒光綠方塊。
- Hover 只改變背景和文字，不做左右晃動。
- 章節序號使用等寬字型，正文名稱保持現代中文字型。
- 摺疊箭頭使用現有圖示庫，不用文本符號。

### 7.3 正文

- H1 更大、更緊湊，標題上方增加 `CHAPTER 01` 畫素標籤。
- H2 前可使用 8px 熒光綠方塊作為章節錨點，但不要每級標題都加裝飾。
- 正文連結使用 Accessible Green，並保留下劃線或明顯 Hover 狀態。
- 引用和提示框使用 Soft Green 背景 + Ink 左邊框，不使用大面積半透明漸變。
- 圖片統一邊框、8px 圓角和輕陰影；有說明時使用小號圖注。
- 程式碼塊使用深色背景，頂部可以有綠色檔名/語言標籤。
- 表格表頭使用 Ink 黑底或淺綠色底，移動端允許橫向滾動。
- 上一篇/下一篇卡片採用硬邊編輯式佈局，明確顯示章節號和標題。

### 7.4 右側目錄

- 標題改為 `ON THIS PAGE / 本頁目錄`。
- 當前小節使用 Ink 粗體 + 綠色豎線。
- 二級、三級標題要有明顯縮排，不只依靠字號。

### 7.5 搜尋

- 保留 VitePress Local Search。
- 搜尋按鈕改為直角感更強的緊湊控制元件，顯示 `⌘ K` 僅作為鍵盤提示。
- 搜尋彈窗中的匹配關鍵詞用淺綠色高亮，選中項用 Ink 黑底。
- 不新增遠端搜尋服務。

### 7.6 深色模式

- 深色模式更接近“編輯臺”，而不是霓虹駭客終端。
- 正文使用暖白文字，綠色只做當前狀態、連結、按鈕和程式碼標籤。
- 不新增 CRT 掃描線、發光字、閃爍游標或 Glitch 效果。

## 8. 動效規範

動效必須短、穩、剋制。

- Hero 初次進入：標題、說明、按鈕依次淡入並上移 `8–12px`，總時長不超過 `500ms`。
- 畫素卡片：首次進入做一次輕微錯位復位；Hover 時旋轉回 `0deg` 並上移 `3px`。
- 主要按鈕：Hover 時硬陰影從 `0` 增加到 `3px 3px 0`，按下時回落。
- 閱讀卡片：Hover 上移 `3px`，路徑線變綠。
- 當前目錄狀態：顏色在 `150–180ms` 內變化。
- 不使用持續漂浮、自動跑馬燈、滑鼠跟隨、視差滾動和大面積滾動觸發動畫。
- 完整支援 `prefers-reduced-motion: reduce`，關閉非必要動畫。

## 9. 響應式規則

### ≥ 1280px

- 內容最大寬度 `1280px`。
- Hero 7:5 兩列，H1 72–88px。
- 四段閱讀路線為 2×2。
- 任務入口為 3×2。

### 768–1279px

- Hero 6:5 或上下佈局。
- 右側畫素卡片減少為 2–3 張。
- 文件頁右側目錄可以隱藏，左側欄保持。

### < 768px

- Hero 單列，左右內邊距 20–24px。
- H1 42–50px，不能出現單字孤行。
- 只保留 2 張畫素卡片，放在標題之後或 Hero 底部，避免遮擋文字。
- CTA 可換行，但主要按鈕優先佔據完整一行。
- 閱讀路線改為單列；卡片高度由內容決定，不設過高最小高度。
- 文件側邊欄改為抽屜，右側目錄隱藏。
- 所有按鈕和圖示觸控區至少 44px。

重點測試寬度：`1440 / 1024 / 768 / 390px`。

## 10. 無障礙與可讀性底線

- 黑字/白字與背景至少滿足 WCAG AA 對比度。
- 熒光綠不承擔白底上的小字號正文。
- 所有互動元素必須有清晰的鍵盤 Focus 狀態，建議使用 `3px` Ink/Acid 雙層焦點環。
- 導航、章節標題和按鈕不能只依靠顏色區分狀態。
- 畫素字型不用於長文，也不用於 14px 以下的中文。
- 圖示提供 `aria-label` 或配套可見文字；純裝飾圖示使用空 alt。
- 深色模式下程式碼、表格、連結和提示框仍要保持清晰層級。
