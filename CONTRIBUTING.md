# 貢獻指南

感謝你願意一起完善《WorkBuddy 實戰藍皮書》。我們希望每次貢獻都能幫助讀者更快完成一個真實任務，並留下可以驗證、複用和遷移的方法。

## 貢獻型別

- 修正錯別字、失效連結、過時截圖或不準確表述。
- 補充真實案例、失敗排查和驗收標準。
- 新增 Skill、聯結器、API、自動化或多 Agent 實踐。
- 增加崗位路線圖和行業工作流。
- 改進網站體驗、搜尋、導航、響應式佈局和無障礙。
- 提供英文翻譯或其他語言版本。

## 內容原則

1. **可復現**：儘量寫清環境、輸入、步驟、交付物和預期結果。
2. **可驗證**：說明怎樣判斷任務真的完成，而不是隻描述過程。
3. **事實與經驗分開**：產品事實附來源；個人經驗說明適用範圍。
4. **安全優先**：涉及檔案寫入、賬號、金鑰、釋出或外部操作時說明許可權與風險。
5. **尊重隱私和版權**：不要提交個人資料、訪問憑據或無權公開的內容。

## 修改正文

每個頁面位於 `docs/bluebook/` 的對應目錄中：

- `index.md`：網站展示的正文。
- `assets/`：該頁使用的圖片、GIF、影片或附件。
- `source.md`、`source.xml`、`metadata.json`：本地同步產生的原始資料，預設不提交到 Git。

請優先修改 `index.md`。新增素材時使用有意義的檔名，並儘量壓縮體積，但不要犧牲可讀性。

## 提交社群 Case

真實的 WorkBuddy 實踐案例請優先提交到：

```text
docs/cases/submissions/<case-slug>/
├── index.md
└── assets/
```

複製 `.github/CASE_TEMPLATE.md` 作為正文起點，並在建立 PR 時使用 `.github/PULL_REQUEST_TEMPLATE/case.md`。案例必須描述場景、使用的 Skills、操作過程、實際效果、驗收標準和安全邊界。

案例集合首頁會在構建時自動讀取這些目錄，不需要手工修改 `docs/cases/index.md`。完整說明見網站的 [Case 投稿指南](https://workbuddy.homes/community/case-contributing)。

## 本地驗證

```bash
npm install
npm run docs:build
npm run docs:preview
```

提交前請確認：

- 構建成功。
- 新增或修改的內部連結可訪問。
- 圖片和影片能載入。
- 手機和桌面寬度下內容都可讀。
- 沒有提交金鑰、個人資訊或構建產物。

## Pull Request

PR 描述請包含：

- 改了什麼。
- 為什麼需要修改。
- 如何驗證。
- 是否涉及產品事實更新；如果涉及，請附來源和核對日期。

儘量讓一個 PR 只解決一個主題，便於審閱和回退。
